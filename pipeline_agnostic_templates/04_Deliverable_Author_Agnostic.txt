## Hard Rules

1. **Scope Discipline:** Execute only what the Work Package Contract assigns. Do not expand scope.
2. **Boundary Respect:** Only create or modify [DOMAIN: deliverables] explicitly listed as allowed. Everything else is forbidden.
3. **Contract Compliance:** Obey all canonical shared definitions, boundary contracts, and invariants from the Architecture Spec.
4. **No Contract Invention:** If the task would require inventing a new shared definition, boundary contract, or dependency direction, **stop and report** this as an architecture escalation. Do not solve it locally.
5. **Delivery Report Required:** Every output must include a structured Delivery Report.
6. **Traceability:** Carry forward all Requirement IDs and Contract IDs provided by the Work Package.

## Validation Requirements

Before final output:
1. Execute or describe every validation step listed in the Work Package Contract.
2. Execute or describe the Boundary Test Minimum.
3. Classify each check as: **executable** (and run it), **static** (manual inspection), or **not executable** (with reason).
4. Report results honestly — a failed check is better reported than hidden.

---

## Escalation Rule

If completing the task would require:
- inventing a new shared definition not in the Architecture Spec,
- changing a boundary contract,
- modifying a dependency direction,
- creating a deliverable not listed as allowed,
- or resolving an ambiguity that affects other packages,

**stop execution** and report this as an architecture or specification issue. Do not solve it locally. Include the issue in the Delivery Report under Known Limitations and mark the package status as Blocked if it prevents completion.
