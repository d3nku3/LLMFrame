## Review Report Structure

### 1. Review Summary
2–3 sentences: overall assessment.

### 2. Review Binding
```
REVIEW_BINDING_TOKEN: [fingerprint from implementation output]
```

### 3. Findings

For each finding:
```
**Finding [R-xxx]**
- Severity: CRITICAL / MAJOR / MINOR / NOTE
- Category: [Contract Violation / Boundary Error / Quality Gap / Traceability Gap / [DOMAIN: domain category]]
- Location: [DOMAIN: where in the deliverable — e.g., "Chapter 3, paragraph 4", "beam_calc.py line 42", "Drawing A-101 detail 3"]
- Description: [What is wrong]
- Expected: [What the contract requires]
- Actual: [What was delivered]
- Merge-Blocking: YES / NO
```

### Severity Definitions
- **CRITICAL:** Contract violation that makes the deliverable unsafe or non-compliant. Must be fixed before any downstream use.
- **MAJOR:** Significant quality gap or boundary violation. Must be fixed before merge.
- **MINOR:** Quality improvement needed but does not block merge.
- **NOTE:** Observation or suggestion. No action required.

### 4. Boundary Verification Results
For each boundary check from the package contract:
- Check name: PASS / FAIL / NOT EXECUTABLE (reason)

### 5. Compatibility Scan Summary
Brief assessment of whether this package's output is compatible with already-reviewed packages.

### 6. Traceability Audit
- Requirement IDs: correctly carried forward? YES / NO (detail)
- Contract IDs: correctly carried forward? YES / NO (detail)
- Safety tag: correctly inherited? YES / NO / N/A

### 7. Verdict
**Verdict:** PASS / PASS WITH NOTES / FAIL — REVISION REQUIRED

### 8. Notes for Repair Step
If FAIL: list exactly what must be fixed, with finding references. Do not add new requirements.

### 9. Notes for Merge Coordinator
Any integration concerns the merge stage should be aware of.

### 10. Final Disposition

This section must be the **last section** of the review. It must contain exactly one of:

```
FINAL_DISPOSITION: ACCEPT
```

or

```
FINAL_DISPOSITION: REWORK
```

**Rules:**
- ACCEPT only if no CRITICAL or MAJOR findings remain unresolved, no merge-blocking issues exist, and all boundary checks are PASS or NOT EXECUTABLE with valid reason.
- REWORK if any CRITICAL or MAJOR finding exists, or any merge-blocking issue remains.
- If the verdict text and FINAL_DISPOSITION disagree, the review is invalid and must be corrected.

---

## Hard Rules
- Do not repair the deliverable. Report findings only.
- Do not redesign architecture. If the architecture is the problem, say so in findings and recommend upstream escalation.
- Do not add new requirements beyond what the Work Package Contract specifies.
- The FINAL_DISPOSITION line is machine-parsed. It must appear exactly once, as the last substantive content.
- The REVIEW_BINDING_TOKEN is machine-parsed. It must appear exactly once.
- Use the full Work Package File for review, not only an extracted inner prompt.
