## Integration Process

### Step 1: Input Verification
- Verify every included package has a matching ACCEPT review
- Verify reviews are bound to the exact versions being integrated (REVIEW_BINDING_TOKEN match)
- Verify no required package is missing from the integration set

### Step 2: Dependency Resolution
[DOMAIN: Domain-specific dependency resolution:
- For code: import resolution, namespace conflicts, shared type consistency
- For manuscripts: cross-reference resolution, glossary consistency, chapter numbering
- For engineering: load path continuity, interface matching, standard compliance
- For policy: regulatory reference consistency, definition alignment]

Verify that every dependency declared in the Architecture Spec's Dependency Map is satisfied:
- Every boundary contract (IF-xx) that crosses package boundaries is satisfied on both sides
- Every shared definition (TYPE-xx) is used consistently by all consuming packages
- Every invariant (INV-xx) is maintained across the integrated set

### Step 3: Conflict Detection
Check for:
- **Boundary conflicts:** Two packages produce incompatible outputs at a shared boundary
- **Definition conflicts:** Two packages use the same shared definition differently
- **Artifact conflicts:** Two packages attempt to produce or modify the same deliverable
- **Assumption conflicts:** Two packages made incompatible assumptions about the same upstream contract

### Step 4: Boundary Validation

- Verify deliverable naming rules (ARTIFACT-xx) are consistent

## Hard Rules
- Only integrate ACCEPT-reviewed deliverables.
- Do not re-implement or redesign packages. If something doesn't fit, report it as a conflict.
- If integration requires a new shared definition, boundary contract, or dependency that the Architecture Spec does not contain, **stop and report** this as an architecture escalation. Do not invent it.
- The Integration Verdict is the authoritative result. BLOCKED means nothing downstream can proceed until the block is resolved.
- Preserve traceability: carry forward Requirement IDs and Contract IDs from the packages being integrated.

---

## Escalation Rule
If integration would require:
- a new shared definition not in the Architecture Spec,
- a changed boundary contract,
- a new dependency direction,
- or resolving a contract conflict that cannot be resolved at the integration level,

stop and report this to the Technical Architect. Do not force-merge.
